## 🟡 Major Issues

### M-1: DEBUG flags declared in Config but not wired into game logic
**File:** `src/shared/Config.lua:123–125` + all server modules

**Finding:**
```lua
Config.DEBUG_INSTANT_HARVEST   = false
Config.DEBUG_FREE_ROLLS        = false
Config.DEBUG_LOG_ROLLS         = false
```
These three flags are declared in Config as documented in TECHNICAL_SPEC §11, but **none of them are read anywhere in the server code**. `FarmManager` never checks `DEBUG_INSTANT_HARVEST` to skip grow timers. `GameManager` never checks `DEBUG_FREE_ROLLS` before deducting coins. `RNGManager` never checks `DEBUG_LOG_ROLLS` to print results.

The flags exist but have zero effect — they mislead developers into thinking debug mode is functional when it isn't. A developer enabling `DEBUG_INSTANT_HARVEST = true` will see no change and waste time debugging why.

**Risk:** Developer confusion, wasted testing time. No player-facing impact since all flags default to `false`.

**Fix:**
```lua
-- In FarmManager.IsReady():
if Config.DEBUG_INSTANT_HARVEST then return true end

-- In GameManager RequestRoll handler, before coin deduction:
if not Config.DEBUG_FREE_ROLLS then
    data.coins -= Config.ROLL_COST_COINS
end

-- In RNGManager.RollSeed(), after picking the seed:
if Config.DEBUG_LOG_ROLLS then
    print(string.format("[RNG DEBUG] luck=%d → rarity=%s seed=%s", luckBonus, result.rarity, result.seedId))
end
```

### M-2: `GetUpgradeCost` RemoteFunction has no input validation
**File:** `src/server/GameManager.server.lua:481–488`

**Finding:**
```lua
RF[RemoteEvents.FunctionNames.GetUpgradeCost].OnServerInvoke = function(_player, payload: {stat: string, level: number})
    if payload.stat == "luck" then
        return CalcLuckUpgradeCost(payload.level)     -- payload.level unchecked
    elseif payload.stat == "harvestSpeed" then
        return CalcHarvestSpeedUpgradeCost(payload.level)
    end
    return 0
end
```
`payload` is assumed to be a table with `stat` (string) and `level` (number). An exploiter can fire `GetUpgradeCost` with `payload = nil`, `payload = "string"`, or `payload = {stat="luck", level=math.huge}`. With `level = math.huge`, `CalcLuckUpgradeCost` returns `math.huge` — while this is not directly exploitable (it's a read-only info call), it could cause the client to display `∞` cost and break the upgrade UI logic. With `payload = nil`, this crashes the handler with an unprotected error.

**Risk:** Unhandled crash in RF handler, potential UI breakage with crafted values.

**Fix:**
```lua
RF[RemoteEvents.FunctionNames.GetUpgradeCost].OnServerInvoke = function(_player, payload)
    if type(payload) ~= "table" then return 0 end
    if type(payload.stat) ~= "string" then return 0 end
    if type(payload.level) ~= "number" then return 0 end
    local level = math.clamp(math.floor(payload.level), 0, 100)
    if level ~= level then return 0 end  -- NaN guard
    if payload.stat == "luck" then
        return CalcLuckUpgradeCost(level)
    elseif payload.stat == "harvestSpeed" then
        return CalcHarvestSpeedUpgradeCost(level)
    end
    return 0
end
```

### M-3: `HasGamepass` RemoteFunction passes passId directly to MarketplaceService without validation
**File:** `src/server/GameManager.server.lua:490–494`

**Finding:**
```lua
RF[RemoteEvents.FunctionNames.HasGamepass].OnServerInvoke = function(player, passId: number)
    local ok, owns = pcall(function()
        return MarketplaceService:UserOwnsGamePassAsync(player.UserId, passId)
    end)
    return ok and owns
end
```
`passId` is type-annotated as `number` but not validated. An exploiter can send `passId = 0`, `passId = -1`, `passId = math.huge`, or `passId = nil`. With `pcall`, these won't crash the server — `UserOwnsGamePassAsync` will return false or error silently. However, negative/zero IDs trigger an API call to Roblox's servers unnecessarily, and a flood of requests with invalid IDs could hit rate limits. More concerning: `passId = nil` bypasses the type annotation at runtime in unstrict contexts.

**Risk:** API request pollution, minor rate limit exposure.

**Fix:**
```lua
RF[RemoteEvents.FunctionNames.HasGamepass].OnServerInvoke = function(player, passId)
    if type(passId) ~= "number" or passId <= 0 or passId ~= passId then return false end
    passId = math.floor(passId)
    local ok, owns = pcall(function()
        return MarketplaceService:UserOwnsGamePassAsync(player.UserId, passId)
    end)
    return ok and owns
end
```

**File:** `src/server/GameManager.server.lua:476–479`

RF[RemoteEvents.FunctionNames.GetSeedInfo].OnServerInvoke = function(_player, seedId: string)

## 💡 Suggestions

### S-1: `WithData` helper utility still not implemented
The suggestion from the previous report (BugByte 2026-05-15 04:40) to create a `WithData(player, fn)` helper wrapper remains unimplemented. 8 out of 10 remote handlers repeat the same `GetData` + nil-check pattern. A helper would make all handlers more uniform and prevent future regressions where someone forgets the nil check.

```lua
local function WithData(player: Player, fn: (data: DataManager.PlayerData) -> ())
    local data = DataManager.GetData(player)
    if data then fn(data) end
end
```

### S-2: `vipLuckApplied` flag would make B-1 fix more robust
The current B-1 fix (idempotent recompute) is correct but relies on `luckLevel` being consistent at all times. If a future change to the upgrade flow inadvertently modifies `luck` directly without touching `luckLevel`, the recompute would silently produce wrong values. Persisting a `vipLuckApplied: boolean` in PlayerData would be a belt-and-suspenders guard.

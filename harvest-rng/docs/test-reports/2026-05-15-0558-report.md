## ­čö┤ Blockers

### B-NEW-1: `SeedData.SeedData[id]` nil dereference ÔÇö crashes client on any plant/roll/timer

**Severity:** ­čö┤ Blocker ÔÇö the core visual loop is broken from the first roll onward
**File:** `src/client/MainClient.client.lua` lines 198, 251, 258, 395

**Finding:**

`SeedData.lua` returns the raw seed table directly:
```lua
local SeedData: {[string]: SeedDefinition} = {}
-- seeds added as SeedData["wheat"] = {...}
-- utilities added as function SeedData.Get(id) / SeedData.GetByRarity(rarity)
return SeedData
```

All correct server-side usages access the returned table directly:
```lua
local SeedData = require(game.ReplicatedStorage.Shared.SeedData)
SeedData.Get("wheat")           -- correct
SeedData.GetByRarity("Common")  -- correct
```

However, `MainClient.client.lua` requires the same module on line 31, then uses `SeedData.SeedData[seedId]` at four different call sites:

| Line | Context | Impact |
|------|---------|--------|
| 198 | `RefreshPlotFrame()` ÔÇö plot icon/rarity colour | Crash on any planted plot |
| 251 | `RollResult` single-roll handler ÔÇö emoji display | Crash on every roll result |
| 258 | `RollResult` x10 handler ÔÇö emoji per result | Crash on every x10 result |
| 395 | Timer loop ÔÇö harvest time calculation | Crash kills entire timer task |

`SeedData.SeedData` evaluates to `SeedData["SeedData"]`. There is no key `"SeedData"` in the returned table ÔÇö keys are seed IDs (`"wheat"`, `"golden_apple"`, etc.) and the three utility functions `Get`, `GetByRarity`, `GetAll`. Accessing an absent key returns **nil**, and the subsequent subscript `nil[seedId]` is a Luau runtime error that terminates the calling coroutine.

**Crash cascade (in execution order):**
1. `PlotStateUpdate.OnClientEvent` Ôćĺ `RefreshPlotFrame()` Ôćĺ crashes Ôćĺ plot icons/colours never update; plots stuck on default state
2. `RollResult.OnClientEvent` Ôćĺ `isRolling = false` runs (line 248) but crashes before inventory refresh (line 263) Ôćĺ roll animation never plays
3. Timer `task.spawn` loop Ôćĺ crashes on first iteration that has a planted seed Ôćĺ ALL countdown timers freeze; "Ôťů Ready!" never appears; harvest buttons never show

A player who rolls any seed and plants it will see no emoji on plots, no roll animation, and frozen timers. The core loop is visually broken.

`InventoryManager.lua` uses `SeedData.Get()` correctly and is not affected.

**Fix ÔÇö replace all four usages with direct table access:**
```lua
-- Lines 198, 251, 258, 395: replace SeedData.SeedData[id]
-- WRONG:
local seedDef = SeedData.SeedData[state.seedId]
-- CORRECT (direct index ÔÇö seed IDs are top-level keys):
local seedDef = SeedData[state.seedId]
-- Or with safety (matches the pcall pattern used in InventoryManager):
local ok, seedDef = pcall(SeedData.Get, state.seedId)
```

### M-NEW-1: `isRolling` never reset on server rejection ÔÇö roll button permanently locked

**Severity:** ­ččí Major ÔÇö triggered every time a player runs out of coins
**File:** `src/client/MainClient.client.lua` lines 79, 248, 334ÔÇô335, 341ÔÇô342

**Finding:**

`isRolling` is set `true` when either roll button is pressed (lines 335, 342) and is only reset to `false` inside the `RollResult.OnClientEvent` handler (line 248):

```lua
-- Client: sets lock on button press
RollButton.Activated:Connect(function()
    if isRolling then return end   -- guarded
    isRolling = true               -- locked
    RE[...RequestRoll]:FireServer()
end)

-- Only unlock point:
RE[...RollResult].OnClientEvent:Connect(function(results)
    isRolling = false              -- unlocked HERE and only HERE
    ...
end)
```

When the server rejects the roll (insufficient coins, data not yet loaded), it fires `Notification` and immediately returns without ever firing `RollResult`:

```lua
-- Server (GameManager.server.lua ~line 248):
if data.coins < Config.ROLL_COST_COINS then
    RE[...Notification]:FireClient(player, { message = "Not enough coins!", style = "error" })
    return   -- no RollResult
end
```

Result: `isRolling` is never reset. The roll button is permanently disabled for the entire session. The player must rejoin to roll again.

This is particularly punishing: new players start with 250 coins (5 rolls), and after emptying their wallet while waiting for a harvest, one rejected roll attempt bricks the button with only a generic toast notification as feedback. The same deadlock applies to `RequestRollX10`.

**Fix:**
Reset `isRolling` inside the `Notification.OnClientEvent` handler for `style == "error"`:

```lua
RE[RemoteEventsModule.Names.Notification].OnClientEvent:Connect(function(payload: {[string]: any})
    if payload.style == "error" then
        isRolling = false   -- unblock roll button on server-side rejection
    end
    UIManager.ShowNotification(payload.message :: string, payload.style :: string?)
end)
```

### N-NEW-1: Client timer omits rarity harvest-time modifier ÔÇö display drift up to 48 min on Mythic seeds

**File:** `src/client/MainClient.client.lua` line 397

**Finding:**

Server formula (`RNGManager.CalcHarvestTime`):
```lua
return math.ceil(seed.harvestTime * rarityMod / speed)
-- rarityMod: Common 1.00, Uncommon 1.00, Rare 0.95, Epic 0.90, Legendary 0.85, Mythic 0.80
```

Client timer loop:
```lua
local effectiveTime = math.ceil(baseTime / math.max(harvestSpeed, 0.1))
-- rarityMod absent
```

Drift at no speed upgrade (harvestSpeed = 1.0):

| Rarity | rarityMod | Server effective time | Client displays | Display drift |
|--------|-----------|----------------------|-----------------|---------------|
| Common / Uncommon | 1.00 | baseTime | baseTime | 0s |
| Rare | 0.95 | 5% shorter | baseTime | up to +15s |
| Epic | 0.90 | 10% shorter | baseTime | up to +90s (+1.5 min) |
| Legendary | 0.85 | 15% shorter | baseTime | up to +540s (+9 min) |
| Mythic | 0.80 | 20% shorter | baseTime | up to +2,880s (+48 min) |

(Mythic worst-case uses Genesis Seed 14,400s base.) For Mythic seeds the client timer shows 4h00m when the server considers the crop ready at 3h12m. The "Ôťů Ready!" state and harvest button never appear on the client until the full drift window elapses. This is most impactful for the rarest and highest-value seeds.

**Fix:** Add rarity modifier to the client timer calculation:
```lua
-- MainClient.client.lua ~line 394
local RARITY_TIME_MOD: {[string]: number} = {
    Common=1.00, Uncommon=1.00, Rare=0.95, Epic=0.90, Legendary=0.85, Mythic=0.80
}
local seedDef       = SeedData[state.seedId :: string]    -- also fixes B-NEW-1
local baseTime      = seedDef and seedDef.harvestTime or 60
local rarityMod     = seedDef and (RARITY_TIME_MOD[seedDef.rarity] or 1.0) or 1.0
local effectiveTime = math.ceil(baseTime * rarityMod / math.max(harvestSpeed, 0.1))
```
Alternatively move `RARITY_HARVEST_TIME_MODIFIERS` into `Config.lua` to share a single source of truth between server and client.

### S-NEW-1: Leaderboard name-lookup hits up to 100 DataStore reads per player request

**File:** `src/server/GameManager.server.lua` ÔÇö `RequestLeaderboard` handler

The handler calls `playerNameStore:GetAsync(entry.key)` for each of up to 100 leaderboard entries per request. With 50 concurrent players opening the leaderboard, this fires ~5,000 DataStore reads in a short window against `PlayerNames_v1`. Roblox's shared DataStore budget at 50 players is ~560 requests/min, so a burst open would hit rate limits.

**Recommendation:** Build a server-side in-memory `userId Ôćĺ displayName` cache (a Lua table populated in `OnPlayerAdded` from the existing `GetNameFromUserIdAsync` call). Resolve names from this cache during leaderboard render, falling back to DataStore only for offline players. This makes leaderboard renders zero-cost for online players.

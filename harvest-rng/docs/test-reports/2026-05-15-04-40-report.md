## 🔴 Blockers (must fix before release)

### B-1: VIP Luck stacks on every rejoin — unlimited Luck exploit
**File:** `src/server/GameManager.server.lua:141`

**Finding:**
```lua
if HasGamepass(player, "VIPPlot") then
    data.luck += Config.VIP_LUCK_BONUS   -- ← += accumulates on EVERY join
```
`data.luck` is loaded from DataStore, then `VIP_LUCK_BONUS` (+15) is **added on top** every time the player joins. A VIP player who joins 7 times reaches Luck 105 (exceeding the max of 100). After ~20 rejoins they're at Luck 300, making `BuildWeights()` produce negative weight for Common (clamped to 1.0 but weight math breaks). This is a **confirmed economy exploit** — a VIP player can grind rejoins for near-guaranteed Legendary/Mythic rolls.

**Risk:** Economy destruction. VIP players can achieve ~100% Legendary+ roll rates within minutes by exploiting rejoin.

**Fix:**
```lua
-- Instead of += (which accumulates), compute luck from scratch:
if HasGamepass(player, "VIPPlot") then
    -- Apply VIP perks only if not already applied (guard with a flag, or recompute from level)
    data.luck = (data.luckLevel * Config.LUCK_PER_UPGRADE) + Config.VIP_LUCK_BONUS
    -- ↑ Always recomputes correctly regardless of how many times player joins
    for i = Config.STARTING_PLOTS + 1, Config.STARTING_PLOTS + Config.VIP_EXTRA_PLOTS do
        if data.plots[i] and not data.plots[i].isUnlocked then
            data.plots[i].isUnlocked = true
        end
    end
    DataManager.MarkDirty(player)
end
```
Alternatively, never save `luck` directly — always derive it at runtime as `luckLevel * LUCK_PER_UPGRADE + (hasVIP and VIP_LUCK_BONUS or 0)`.

### B-2: No rate limiting on RequestRoll / RequestRollX10 — coin drain race condition
**File:** `src/server/GameManager.server.lua:215–253`

**Finding:**
An exploiter can fire `RequestRoll` thousands of times per second via a script. While the coin check (`data.coins < ROLL_COST_COINS`) exists, the handler does **not use `pcall` around the mutation block** and has **no per-player cooldown**. Under rapid fire, Luau's cooperative scheduler could theoretically allow two concurrent handler threads to both pass the `data.coins < 50` check before either deducts — a classic TOCTOU (time-of-check-time-of-use) race. In Roblox's single-threaded server, this specific race is less likely but remote spam still has a real cost: it floods the server's event queue, burns DataStore dirty marks, and causes excessive `FireClient` calls.

**Risk:** Server performance degradation, potential double-spend under specific scheduling; confirmed remote spam vector.

**Fix:** Add a per-player cooldown table:
```lua
-- Near top of GameManager, after RE/RF creation:
local rollCooldowns: {[number]: number} = {}  -- userId → last roll os.clock()
local ROLL_COOLDOWN = 0.5  -- seconds between rolls

-- In RequestRoll handler, before coin check:
local now = os.clock()
if (now - (rollCooldowns[player.UserId] or 0)) < ROLL_COOLDOWN then return end
rollCooldowns[player.UserId] = now

-- In PlayerRemoving:
Players.PlayerRemoving:Connect(function(player: Player)
    rollCooldowns[player.UserId] = nil
    DataManager.Unload(player)
end)
```

### M-2: `plotIndex` not range-clamped in remote handlers — out-of-bounds array access
**File:** `src/server/GameManager.server.lua:260–298`

**Finding:**
```lua
RE[RemoteEvents.Names.RequestHarvest].OnServerEvent:Connect(function(player, plotIndex: number)
    if type(plotIndex) ~= "number" then return end
    local result = FarmManager.Harvest(player, plotIndex)  -- plotIndex unclamped!
```
Type is checked (`type() == "number"`) but the value is **never range-clamped** before passing to `FarmManager`. An exploiter can fire `RequestHarvest` with `plotIndex = 999999` or `plotIndex = -1`. Inside `GetPlot()`, `data.plots[999999]` returns `nil` gracefully, so no crash — but `data.plots[-1]` or `data.plots[0]` are also `nil` and return early, wasting a server handler invocation. More importantly, passing `math.huge` or `1/0` (NaN) as `plotIndex` bypasses the type check and could produce undefined behaviour in array indexing.

**Fix:** Add bounds clamp to all three plot-index handlers:
```lua
-- Add after each type check:
plotIndex = math.clamp(math.floor(plotIndex), 1, Config.MAX_PLOTS)
if plotIndex ~= plotIndex then return end  -- NaN guard
```

### M-3: `seedId` string length not bounded — potential DataStore key bloat
**File:** `src/server/GameManager.server.lua:261`

**Finding:**
```lua
if type(plotIndex) ~= "number" or type(seedId) ~= "string" then return end
```
`seedId` is type-checked as a string but **not length-limited**. An exploiter can send a 1MB string as `seedId`. It will fail `SeedData.SeedData[seedId]` lookup (returning nil) in `FarmManager.PlantSeed`, so no planting exploit — but the string is briefly allocated in server memory, and if somehow a path existed to write it to DataStore it would exceed Roblox's 4MB key limit.

**Fix:**
```lua
if type(plotIndex) ~= "number" or type(seedId) ~= "string" or #seedId > 64 then return end
```

### M-4: Daily streak can be claimed unlimited times per session via `RequestClaimStreak`
**File:** `src/server/GameManager.server.lua:396–399`

**Finding:**
```lua
RE[RemoteEvents.Names.RequestClaimStreak].OnServerEvent:Connect(function(player)
    HandleDailyStreak(player)
    SendStatsUpdate(player)
end)
```
`HandleDailyStreak` correctly checks `hoursSince >= 20` before granting rewards, so repeated legitimate calls within the same day are harmless — they just update `lastLogin`. However, there is **no cooldown** on this remote. A bot can fire `RequestClaimStreak` thousands of times per second, burning server event queue budget. Additionally, if `lastLogin` was 0 (brand-new player), `hoursSince` will be `(now - 0) / 3600` ≈ **hundreds of thousands of hours** — so `hoursSince >= DAILY_STREAK_RESET_HOURS (36)` immediately resets streak to 0, then `hoursSince >= 20` immediately grants Day 1 reward. This is correct by logic but means a new player **always gets a day-1 streak reward on their very first join**, even if `HandleDailyStreak` is also called in `OnPlayerAdded`. This means the reward fires **twice on first join** (once from `task.delay(0.5)` in `OnPlayerAdded`, and possibly again if the client fires `RequestClaimStreak` on GUI open).

**Fix:**
```lua
-- Add per-player cooldown:
local streakCooldowns: {[number]: number} = {}
RE[RemoteEvents.Names.RequestClaimStreak].OnServerEvent:Connect(function(player)
    local now = os.clock()
    if (now - (streakCooldowns[player.UserId] or 0)) < 5 then return end
    streakCooldowns[player.UserId] = now
    HandleDailyStreak(player)
    SendStatsUpdate(player)
end)
```
Also in `HandleDailyStreak`, guard the first-login case:
```lua
if data.lastLogin == 0 then
    data.lastLogin = now
    DataManager.MarkDirty(player)
    return  -- don't grant day-1 reward on server-side init; let client claim it
end
```

### M-5: Harvest speed formula mismatches GDD description
**File:** `src/server/GameManager.server.lua:364`, `src/shared/Config.lua:61`

**Finding:**
GDD §5.3 states: *"Each level reduces harvest time by 10% (multiplicative). Max speed ≈ 4.7× faster."*

The formula in GameManager is:
```lua
data.harvestSpeed = Config.STARTING_HARVEST_SPEED / (Config.HARVEST_SPEED_FACTOR ^ data.harvestSpeedLevel)
-- = 1.0 / (0.90 ^ level)
```
At level 15: `1.0 / (0.90^15) = 1.0 / 0.2059 ≈ 4.857×` — the GDD says `≈ 4.7×`.

The description is *close* but the formula is `1 / factor^level` which is `÷ 0.90` per level (×1.111 per level), not `× (1 - 0.10)` per level as the GDD implies. These produce identical results so it's a **doc/code mismatch, not a game-breaking bug**, but it will confuse any developer trying to balance the system.

**Fix:** Update GDD §5.3 to read:
> "Each level divides harvest time by 0.90 (effectively ×1.111 speed per level). At max level (15): ~4.86× base speed."
